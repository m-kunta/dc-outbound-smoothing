## Project in One Sentence

**LevelSet** is a Python/Streamlit app that takes a spiky DC outbound replenishment schedule and produces a flatter, capacity-constrained ship plan — same store service levels, less overtime.

**Status: ✅ Fully built and operational.** All 26 backend tests pass. The Streamlit UI is complete.

## Solver Logic (solver.py) — READ THIS CAREFULLY

### Entry Point

```python
result = solve(db_path, *, horizon_days, frozen_hours, lambda_val, gamma_val)
# Returns: {"plan": pd.DataFrame, "kpis": dict}
# Side effect: writes smoothed_plan table to SQLite
```

### Execution Pipeline

1. **`load_data()`** — Reads all 5 input tables from SQLite into DataFrames.
2. **`convert_units()`** — `QTY_PALLETS = ceil(QTY_CASES / UOM_CONV)`. All capacity comparisons in pallets.
3. **`classify_orders()`** — HARD orders are pinned to `NEED_DATE` with `SHIFT_DAYS=0`. SOFT orders go to `smooth()`.
4. **`smooth()`** — The core algorithm (see below).
5. **`compute_kpis()`** — Before/after CV, OSA %, orders shifted, alerts, cube utilisation.
6. **Write to DB** — `smoothed_plan` table replaced on each run.

### `smooth()` — Core Algorithm

**Trigger:** A day is a smoothing candidate if `load > avg_by_resource[resource] × SMOOTH_PEAK_RATIO (default 1.05)` OR `load > dc_capacity`. This prevents over-optimizing naturally flat days.

**For each SOFT order on a peak day:**
1. Scan candidate dates backward from `NEED_DATE` up to `HORIZON_DAYS`.
2. Filter candidates through 5 guardrail checks (all must pass):
   - **Capacity headroom:** `dc_cap[candidate, resource] - current_load >= pallets`
   - **Inventory readiness (REQ-05):** on-hand ≥ qty_cases OR ASN_ETA ≤ candidate date
   - **Shelf-life (MRSL):** `pull-forward days ≤ max(shelf_life // 4, 1)`
   - **Store calendar (REQ-06):** weekday ∈ store's `DELIVERY_CALENDAR`
   - **Backroom cap (REQ-06):** `current_store_day_load + pallets ≤ BACKROOM_CAP`
3. Score each valid candidate: `flattening_gain - λ_pressure_penalty - γ_early_penalty`
4. Additional filter: destination load after move must be `< source_load × SMOOTH_TROUGH_RATIO (0.90)`.
5. Accept best-scoring candidate or raise `"⚠️ No valid window — capacity alert"`.

### Objective Function (conceptual, implemented as greedy scoring)

```
Z = Σ(V_t − μ_H)² + λ·(OSA_Penalty) + γ·(EarlyShipPenalty)
```

- `V_t` = pallets on day t
- `μ_H` = mean daily volume over horizon
- `λ` = OSA penalty weight (default 100, high = protect HARD orders)
- `γ` = early-ship penalty (default 1, high = less aggressive pull-forward)

### Solver Config Parameters

| ENV Var | Default | Sidebar Control | Meaning |
|---|---|---|---|
| `LAMBDA` | 100 | λ number_input (1–500) | Penalizes moving into days with high HARD load |
| `GAMMA` | 1 | γ number_input (0–20) | Penalizes pulling orders forward many days |
| `FROZEN_HOURS` | 48 | Slider (24–96h) | Orders within this window are locked |
| `HORIZON_DAYS` | 10 | Slider (5–14) | Look-back window for trough search |
| `SMOOTH_PEAK_RATIO` | 1.05 | `.env` only | Trigger smoothing above 105% of avg daily |
| `SMOOTH_TROUGH_RATIO` | 0.90 | `.env` only | Accept trough only if < 90% of source load |

**Important:** `app.py` writes sidebar values directly to `os.environ` so `solver.get_runtime_config()` picks them up at call time. Don't break this pattern.

### Exception Types (app.py lines 466–537)

| Type | Color | Trigger | Impact Formula |
|---|---|---|---|
| 🔴 Capacity Alert | Red | SOFT order with `MOVE_REASON.startswith("⚠️")` | `pallets × urgency × 1` |
| 🟠 Near-Frozen HARD | Orange | HARD order with `SMOOTHED_DATE` ≤ today + 3 days | `pallets × (4−days) × 3` |
| 🟡 Day Over-Capacity | Yellow | Post-smooth day still exceeds DC cap | `overflow_pallets × urgency` |

Urgency = `max(10 − days_until_need, 1)` — imminent exceptions always rank above distant ones.

## Design Decisions & Gotchas

1. **Greedy, not exact.** The solver is a greedy heuristic (backward scan + scoring), not a true MIP/LP. This is intentional — exact solvers are too slow for daily operational use with 2,000+ orders. The "SciPy SLSQP" reference in early docs was a planning artifact; the actual implementation is the greedy score function in `smooth()`.

2. **`avg_by_resource` is critical.** The peak-trigger threshold is `avg × SMOOTH_PEAK_RATIO`. If `avg_by_resource` is missing or zero, no smoothing occurs. It's computed in `solve()` across the full horizon and passed to `smooth()`.

3. **Running loads are mutable.** `smooth()` maintains `running_load` and `store_day_load` dicts that update as each order is moved. Order processing sequence matters — orders are processed `NEED_DATE` descending (farthest first). Do not change this.

4. **`os.environ` pass-through.** Sidebar solver params are written to `os.environ` before `solve()` is called. `get_runtime_config()` reads from env. Don't bypass this; the test suite also sets env vars in `setup_module()`.

5. **`levelset.db` is git-ignored.** Always run `python data_gen.py` first after cloning. Tests auto-generate `test_levelset.db` and clean it up.

6. **`lru_cache` on `_get_weekday`.** Cache is unbounded — if dates span > a year, clear with `_get_weekday.cache_clear()` (not an issue in current 30-day horizon).

7. **Exception scoring in `app.py`.** The impact score formula is in `app.py` (not `solver.py`) and is purely for UI ranking — it does not affect the solver output. Don't move it to `solver.py`.

## Common Tasks

### Add a new guardrail to `smooth()`
1. Add a helper function following the `def guardrail_ok(...) -> bool:` pattern.
2. Add a `if not guardrail_ok(...): continue` inside the candidate loop (before the score check).
3. Add a test in `test_backend.py` following the `_make_order` / `smooth()` direct-call pattern.

### Change the scoring function
Edit lines ~270–278 in `solver.py`. The score must be `> 0` for a move to be accepted. Rerun `pytest test_backend.py -v -k "lambda or gamma"` to confirm λ/γ tests still pass.

### Add a new KPI
1. Compute it in `compute_kpis()` and add to the returned dict.
2. Add a `st.metric()` card in `app.py` — expand the `st.columns(5)` to `st.columns(6)` if needed.
3. Add to `kpis_to_json()` in `data_loader.py` if it should be exported.

### Add a new exception type in the UI
Follow the pattern in `app.py` lines 466–537: build a list of dicts with the standard exception schema, append to `exceptions`, the rest of the rendering is generic.

### Add a new CSV upload table
Add an entry to `TABLE_SCHEMAS` in `data_loader.py`. The UI upload loop in `app.py` iterates `TABLE_SCHEMAS` automatically.
